PHASE 2 — ROLES (ADMIN VS USER)
------------------------------------------------------------

Goal:
Separate privileged users from normal users, with a clear authority hierarchy suitable for a production ChatGPT-like system.

To Build:
— Role system on users:
    — user — normal chatbot access
    — admin — manage users
    — owner — full system authority (admins cannot override)

— Role-based access control (RBAC):
    — Admin/owner-only API routes
    — Hard 403 for unauthorized access

— Admin management endpoints:
    — List users (pagination + filters)
    — View single user
    — Update user (role, plan, active status)
    — Manually verify email

— Safety rules:
    — Admins cannot modify owners
    — Only owners can grant/revoke owner role
    — System must never allow removing the last owner
    — System must never allow removing the last admin

— Audit logging (non-negotiable for production):
    — Log every admin action (success + denied)
    — Capture actor, target, action, before/after, IP, user agent

— Promotion method:
    — Manual DB update (bootstrap)
    — Admin API actions (post-bootstrap)

— Exit Criteria:
    — Non-admins receive 403 on admin routes
    — Admins can manage users but not owners
    — Owners can manage everything
    — All admin actions are audit-logged

PHASE 4 — USAGE LIMITS & ANTI-ABUSE
------------------------------------------------------------

Goal:
Protect infrastructure and AI costs (production MVP).

What we built (production MVP):

A) Usage limits (DB-backed, per user, per plan)
- plan_entitlements limits (nullable ints):
  - daily_messages_limit
  - monthly_messages_limit
  - per_minute_messages_limit
- Usage counters table (Postgres):
  - usage_counters(user_id, period_type, period_start, messages_used)
  - period_type values: "minute", "day", "month"
  - period_start stored as naive UTC timestamps
- Quota enforcement (server-side):
  - enforce_mode_and_quota() locks rows with SELECT ... FOR UPDATE
  - Checks BEFORE increment:
    - per-minute limit => HTTP 429 "Per-minute message quota exceeded."
    - daily limit     => HTTP 429 "Daily message quota exceeded."
    - monthly limit   => HTTP 429 "Monthly message quota exceeded."
  - Increments minute/day/month counters on allowed requests
  - Admin/Owner bypasses quotas

B) Input size limits
- Request schema validation:
  - ChatRequest.message min_length=1
  - ChatRequest.message max_length=4000

C) Abuse logging (DB-backed)
- Abuse events table (Postgres):
  - abuse_events(user_id, email, ip, user_agent, endpoint, method, mode, status_code, reason, detail, created_at)
- Logging triggers:
  - 403 feature not allowed  => reason="feature_not_allowed"
  - 404 thread not found     => reason="thread_not_found"
  - 429 quotas exceeded      => reasons:
    - "per_minute_quota_exceeded"
    - "daily_quota_exceeded"
    - "monthly_quota_exceeded"
  - 500 missing entitlements => reason="missing_entitlements"

D) API caching (DB-backed, Postgres)
- api_cache table (Postgres):
  - api_cache(user_id, mode, request_hash, response_json, expires_at, created_at)
  - Unique constraint:
    - (mode, request_hash, COALESCE(user_id, '0000...'))
- Cacheable modes (MVP):
  - indicators_basic
  - indicators_advanced
- Cache behavior:
  - Stable request hash (mode + normalized message)
  - TTL (default): 30 seconds
  - Response returns: "cache": {"hit": true/false}

E) Thread caps (production MVP safety for DB)
- Per-plan thread caps from environment variables:
  - MAX_THREADS_FREE
  - MAX_THREADS_PLUS
  - MAX_THREADS_PRO
- Enforced server-side on thread creation

Exit criteria (checked)
- Daily quotas enforced ✅
- Monthly quotas enforced ✅
- Per-minute rate limit enforced ✅
- Input length limit enforced ✅
- Abuse events recorded ✅
- API cache written & cache hit works ✅
- Thread caps enforced ✅

Status: DONE ✅

PHASE 5 — CHAT MEMORY SYSTEM (PRODUCTION)
------------------------------------------------------------

Goal:
Make conversations feel continuous, intelligent, and plan-aware.
Memory must live in the database — not in Python globals.

Design Principles:
- Backend is the law
- Memory is explicit, scoped, and controllable
- Different plans get different memory power
- No hallucinated memory
- Everything auditable in DB

------------------------------------------------------------
MEMORY LAYERS
------------------------------------------------------------

1) Short-Term Context Window (ALL PLANS, plan-scaled)
----------------------------------------------------
Purpose:
Maintain conversational continuity inside a thread.

What it is:
- Rolling window of recent messages for a thread
- Loaded from DB on every request
- Injected into model prompt

How it works:
- Messages stored in `messages` table (already exists)
- On /chat:
  - Load last N messages for thread_id
  - N is plan-based:
    - Free:   ~10–20 messages
    - Plus:   ~40–80 messages
    - Pro:    ~120–200 messages
- Old messages remain in DB but are not sent to model

Key rules:
- Memory is per-thread
- No cross-thread leakage
- No global session memory
- Admin/Owner bypass limits

Status: DONE ✅


2) Long-Term Memory Summaries (PAID PLANS ONLY)
-----------------------------------------------
Purpose:
Preserve important context without exploding token usage.

What it is:
- Periodic summarized memory of a thread
- Stored separately from raw messages
- Used to “remind” the model of past context

How it works:
- Background or triggered summarization:
  - Example trigger:
    - every X messages
    - or when token estimate exceeds threshold
- Summary replaces older message blocks in prompt
- Raw messages are never deleted

Plan gating:
- Free: ❌ no long-term memory
- Plus: ✅ per-thread summaries
- Pro:  ✅ per-thread + cross-thread summaries (later)

DB storage:
- New table: `thread_summaries`
  - thread_id
  - summary_text
  - covered_until_message_id
  - updated_at

Key rules:
- Summaries are additive, not destructive
- Summaries are model-generated but human-readable
- Can be inspected/debugged later

Status: DONE ✅


3) Structured Memory (OPT-IN, HIGH VALUE)
-----------------------------------------
Purpose:
Store facts that should persist beyond conversation flow.

Types:
- User preferences (risk level, language, style)
- Watchlists (coins, strategies)
- Saved strategies or rules
- System notes (admin-only)

How it works:
- Explicit save/update actions only
- NEVER auto-inferred silently
- Read-only unless user or system updates

DB storage (planned):
- `user_preferences`
- `user_watchlists`
- `saved_strategies`

Plan gating:
- Free: ❌
- Plus: Limited structured memory
- Pro: Full structured memory

Key rules:
- Structured memory is deterministic
- Not mixed with chat history
- Always injected as a separate system block

Status: NOT STARTED
